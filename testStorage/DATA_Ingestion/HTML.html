RTM No.Requirement Description Remarks Status
9
1The service shall synced automatically Incoming files from SFTP to uploader instance, where they are automatically backed up to s3 (in their original file structure and format).
10
2Bucket to update an SQS Queue shall be setup for each object that is added.
11
3The service shall have all the services detailed below store data that's related to the internal state of the service, in a private mongo database (on the mongo server). Data for each report will be stored in an internal DB until all the expenses have been accumulated, at which time will be sent to the IDB using the 'pacific' service DAL - Data Access Layer)
12
4The service shall have an 'imaginary' service which will store evidence including any metadata that may be required in order to reconciliate the evidence with the related expense/s.
13
5The service shall have incomingfilesservice that will register for messages from the queue
14
6The service shall acknowledged each message to SQS once it's received and stored for actual handling in a collection in the private DB with state 'new' and 'lockTime' set to current time.
15
7The service shall have the files arrive from each customer into a directory called reclaim4xxxx, where xxxx is a name of an account, but for older accounts, it is just an arbitrarily structured name and in the future we'd like to support arbitrary names for the directories. Arbitrarily structured name for Older accounts not yet supported
16
8An 'AccountsSetupBackend' service shall provide an endpoint that will accept the name of a directory and return the following:
ftpAccountId - ID of the FTP Account.
In most cases it may be identical to the companyId, but in special cases where the same account has multiple delivery formats, it will have to be something else.
Either way, the 'AccountsSetupBackend' service will return an ftpAccountId in order to make the origin abstract.
17
9The service shall return an 'unidentified' response if the directory name is not found by the 'AccountsSetupBackend' service.
18
10Depending on the response from the 'accounts' service, this service shall act on 'new' messages, as follows:
- If handlingType is 'incomingFiles' - store the ftpAccountId, recipients, rules; change state to 'readyToProcess' and start handling.
- If handlingType is any other - store the ftpAccountId, recipients; change state to 'skipped'; set the skipReason to the value of 'handlingType'
- if 'unidentified' - change state to 'skipped'; set the skipReason to 'unidentified'. In addition, in this case the service should sent an email to <a role at VATBox responsible for new accounts>
19
11Whenever the service is free to process, it shall search for an unlocked message to handle (in the priority described below), lock it (as described below) and perform the relevant handling

***Searching for unlocked messages refers to searching for a message with a specified status and 'lockTime' more than MaxLockTime seconds ago, unless a different period is explicitly specified.
20
12The service shall have a Locking a message and is done by setting lockTime to the current time.
21
13To achieve the desired locking behavior we MUST use findOneAndUpdate for performing the search and updating the 'lockTime' value in one atomic operation.
22
14Handling priority shall be as below (after each handling described below, return to the start, so for example, all 'new' messages will be handled before searching for 'inProgress' messages)
Message in 'new' state - lock it in 'new' state. Perform the process above to identify the FTP Account and report the message.
Message in 'readyToProcess' state - lock it in 'inProgess' state and perform processing as detailed in the next sections.
Message in 'inProgress' state and lockTime more than MaxIncomingFileMessageHandlingTime minutes ago - lock in 'retry' state and set 'retryCounter' to 0. Perform processing.
Message in 'retry' state and lockTime more than MaxIncomingFileMessageHandlingTime minutes - lock in 'retry' state and increment 'retryCounter'.
If retryCounter MaxIncomingFileRetries, change to 'failed' state (TBD report this somewhere)
If retryCounter < MaxIncomingFileRetries perform processing.
Message in 'finished' state - search only for a message updated more than IncomingFilesArchivalDays days ago and lock it in 'movingToGlacier' state setting retryCounter to 0. Try to move the file to Glacier.
Message in 'movingToGlacier' state - search only for a message updated more than y hours ago and lock it in 'movingToGlacier' state, incrementing retryCounter. If retry counter MaxGlacierRetries - try to move to Glacier. If > MaxGlacierRetries, skip.
If no messages found for handling - go to sleep for PollTasksDelay seconds.
23
15The service shall first rename each file to contain the full path except the customer root (e.g. reclaim4amazon/xxx/yyy/zzz/filename.zip will be renamed to xxx_yyy_zzz_filename.zip).
24
16The System shall have a main handler deciding what to do with the file, depending on its type. Type should be identified using something that possibly takes the extension into account and definitely the contents (e.g. mpmmagic).
Depending on the file type:
Subdirectory - will be processed recursively. Each subdirectory traversed will be added to the filename.
archive (jar, zip, rar, 7z, tar, gz, eml, msg) - will be unpacked into a subdirectory and the archive will be deleted. The subdirectory will be processed recursively. If there's not enough space to open the archive, the object should be transitioned to 'failed' state and a message should be sent to the support engineer.
Supported content formats (see below) - will be converted to png (if single image) or pdf (if multiple pages). jpeg and psd (photoshop) will be left as-is. Note: We have to explore the possibility of using ready-made docker containers to perform the actual conversions E.g. for libreoffice)as much as possible or create containers that implement those conversions that aren't available off-the-shelf.
Supported mega-report formats - will be uploaded and sent to the next service - see below.
25
17While processing a message (inProgress), every once-in-a-while (timebased, as long as the process is progressing) the lockTime of the current message in the DB should be updated, to prevent the garbagecollection from changing the message to 'retry' state. Clarification - this keep-alive should work only if the process is progressing, it shouldn't keep the message locked indefinitely.
26
18The service shall support file formats: doc ppt xls bmp odt - opendocument text odp - opendocument presentation ods - opendocument spreadsheet rtf pptx docx xlsx tiff xml gif html pdf png jpeg psd - photoshop
27
19Uploading the file - Each final evidence file will be uploaded to 'imaginary' using an endpoint. 'imaginary' will return an imaginaryID and an indication whether this is a duplicate.
28
20transactionID - currently there's a transactionID that connects a spreadsheet to the files that came with it. This is intended to allow to search for reportIDs that are assumed not to be globally unique per acct. However, we already know that they ARE globally unique per acct, so we don't need this any more.No need for TransactionID? Can omit this item?
29
21The service shall have a response When the file is uploaded to 'imaginary':
Failed - the upload failed - there should be a retry and if it fails again the file should be uploaded to another s3 bucket and a notification should be sent to the support engineer email listed (if the file is part of an archive, all the notifications for the archive should be consolidated into a single email).
Created - the upload succeeded and this wasn't a duplicate - continue normally.
OK - the upload wasn't completed because this was a duplicate, but the existing imaginaryId is returned and can be used. Since we don't really care why this was a duplicate, this is treated like the previous case - continue normally. However, the service should check whether the previous filename was identical - if it wasn't this should be noted in the data sent below, because it might indicate misusing the same evidence for two different cases.
30
22The the following will be done if the file is uploaded successfully:
If the file is an image/document, a Lambda will be triggered for initiating the matching process for this file. The lambda itself is not in the scope of this project.
31
23If a mega-report is found, it will be uploaded to a new bucket intended for this purpose and a 'megaReport' message posted to SQS queue (can't use the s3 notification mechanism for this because we have additional data to add) together with the dateReceived, ftpAccountId, rules and origObject. This is a notification to the 'expensereportimporter' service.
The service will store in the message a counter of mega-reports encountered.
Once the incoming object has been fully handled (meaning all files inside have been uploaded successfully or unsuccessfully, if it's an archive, or meaning the file itself has been uploaded, if it's just a single file), the status should be switched from 'inProgress' to 'finished'
32
24In case we decide to handle some of the file conversions asynchronously using external services, this will pose a slight problem, because you have to wait for the conversions to complete before you know that handling has completed. Also, in that case, it may be necessary to keep the state of the message 'inProgress' in cases where the conversion services are just busy and take a long time to address these requests. If and when relevant, the mechanism for managing this case will be discussed separately.
